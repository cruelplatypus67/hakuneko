# Git workflow

These rules apply to every agent and human contributor working in this repository.

## Commit after every task

1. **Commit when a unit of work is done** — not at the end of a long session.
   A “task” means: a feature slice, bug fix, script, config change, refactor step, rule file, or other coherent checkpoint that leaves the tree in a sensible state.
2. **Do not accumulate large uncommitted diffs.** If you finished something useful, commit it before starting the next thing.
3. **If a task has multiple natural checkpoints**, make multiple commits (e.g. script first, then wiring in `package.json`).
4. **Never leave the session with uncommitted intentional work** unless the user explicitly said not to commit.

## Commit message style (human, professional)

Write messages a solid engineer would put on a clean PR history:

- **Imperative mood, present tense**: “Add…”, “Fix…”, “Refactor…”, not “Added” / “Fixed” / “I added”.
- **Short subject line** (roughly ≤72 chars). Optional body for *why*, not a diary of *how*.
- **Explain intent**, not implementation noise. Prefer “Cross-package Windows and macOS targets from macOS” over “Updated scripts/package-win-mac.mjs”.
- **No AI attribution**, no `Co-authored-by` for tools, no “Generated by…”, no emoji spam.
- **Match project tone** when history already has a pattern (e.g. `fix: SiteName - detail`); otherwise keep it plain and professional.
- **One logical change per commit.** Don’t mix unrelated fixes.

### Good

```
Add script to package Windows x64 and macOS arm64 builds

Cross-builds the Electron client on macOS, copies artifacts to dist/,
and cleans intermediate redists to save disk space.
```

```
fix: MangaDex - handle empty chapter list
```

### Bad

```
update stuff
```

```
WIP
```

```
Changes by AI assistant for packaging and also git config and more
```

## Staging and safety

- Stage **only** files that belong to the checkpoint (`git add` paths explicitly when possible).
- Never commit secrets, credentials, SSH private keys, or local env files.
- Do not amend published commits unless the user asks.
- Do not force-push unless the user explicitly requests it.
- Do not change **global** Git config from automation; use **repository-local** config only when the user asks.

## After committing

- Prefer small, reviewable commits over one giant dump.
- Push only when the user asks, or when standing project rules for this fork explicitly require it (still never force-push by default).
